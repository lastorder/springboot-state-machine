订单初始化状态机设计文档

1. 状态机总览

本设计使用 Fork/Join 模式实现订单初始化流程，等待工厂系统返回 VOM 和 DOM 事件。状态机使用 orderNo 作为唯一标识，实现外部系统统一使用 orderNo 进行交互。

核心特性：
  • orderNo 绑定：状态机以 orderNo 作为 machineId，外部系统无需关注内部 orderId
  • 订单初始化：PR审批通过后开始本地初始化
  • Fork/Join 模式：VOM 和 DOM 事件并行等待，两者都收到后才完成
  • COE 发送：本地初始化后发送 COE 到 Kafka
  • Deal 同步：提交工厂订单后同步状态到 Deal 服务
  • 失败处理：VOM_FAILED 事件直接导致初始化失败
  • 状态同步：状态机停止时自动同步最终状态到 orders 表

2. 状态机绑定设计

2.1 为什么使用 orderNo 作为 machineId？

对比项 使用 orderId（旧方案） 使用 orderNo（当前方案）
外部系统交互 ❌ 需要先查询 orderId ✅ 直接使用 orderNo
Kafka 消息 ❌ 需要携带 orderId ✅ orderNo 是自然主键
ID 迁移 ❌ 需要复杂的 transferOrInitialize() ✅ 无需迁移
幂等处理 ⚠️ 需要额外检查 ✅ orderNo 天然唯一
调试友好性 ⚠️ ID 不直观 ✅ "ORD-001" 直观可读

2.2 状态同步机制

sequenceDiagram participant K as Kafka participant C as OrderEventConsumer participant SM as StateMachineService participant SML as StateMachineListener participant OSS as OrderStatusSyncService participant DB as orders 表 K->>C: 消息 (orderNo="ORD-001") C->>SM: sendEventByOrderNo("ORD-001", event) SM->>SM: machineId = orderNo SM->>SM: 状态转换执行 SM->>SM: persistStateMachine() SM->>SML: stateMachineStopped(machineId="ORD-001") SML->>OSS: syncOrderStatusByOrderNo(orderNo, finalState) Note over OSS: @Transactional(REQUIRES_NEW) OSS->>DB: UPDATE orders SET status = ? DB-->>OSS: 成功
状态同步关键设计：
  • StateMachineListener.stateMachineStopped() 触发状态同步
  • OrderStatusSyncService 使用 @Transactional(REQUIRES_NEW) 独立事务
  • 状态同步失败不影响状态机执行（catch 异常，仅记录日志）
  • 同步时机：状态机停止时（每次事件处理后）

3. 事务处理策略

3.1 事务边界设计原则

重要：Spring State Machine 不是事务安全的

状态机的 Action 执行、状态持久化、状态同步需要在事务边界上仔细设计。

组件 事务策略 原因
StateMachineService.sendEventByOrderNo() 无事务 状态机操作涉及 Reactive 流，不能在传统事务内执行
PrApprovedAction.execute() 无 @Transactional 由调用方控制事务边界，Action 内部调用带事务的 Repository
OrderRepository.save() 继承调用方事务 Spring Data JPA 默认继承事务
OrderStatusSyncService.syncOrderStatusByOrderNo() @Transactional(REQUIRES_NEW) 独立事务，确保状态同步不受外部事务影响

3.3 为什么使用 REQUIRES_NEW？

OrderStatusSyncService 使用 REQUIRES_NEW 的原因：
  1. 隔离性：状态同步失败不影响状态机持久化
  2. 一致性：即使外部事务回滚，状态同步也已提交
  3. 独立提交：每次状态变更立即同步并提交
  4. 错误处理：catch 异常后仅记录日志，不传播
@Service
class OrderStatusSyncService(
    private val orderJpaRepository: OrderJpaRepository,
) {
    // 独立事务，确保状态同步不受外部事务影响
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    fun syncOrderStatusByOrderNo(orderNo: String, newStatus: OrderStatus): Boolean {
        val entity = orderJpaRepository.findByOrderNo(orderNo) ?: return false
        entity.status = newStatus
        entity.updatedAt = Instant.now()
        orderJpaRepository.save(entity)
        return true
    }
}

// Listener 中调用
override fun stateMachineStopped(stateMachine: StateMachine<OrderStatus, OrderEvent>) {
    val orderNo = stateMachine.id  // machineId = orderNo
    val finalState = stateMachine.state?.id
    
    try {
        orderStatusSyncService.syncOrderStatusByOrderNo(orderNo, finalState)
    } catch (e: Exception) {
        // 状态同步失败不影响状态机执行，只记录日志
        log.error("Status sync failed but state machine continues", e)
    }
}

3.4 幂等处理（PrApprovedAction）

override fun execute(context: StateContext<OrderStatus, OrderEvent>) {
    val orderNo = context.message.headers.get("orderNo") as? String ?: return
    
    // 检查订单是否已存在
    val existingOrder = orderRepository.findByOrderNo(orderNo)
    
    if (existingOrder != null) {
        // 幂等检查：获取当前状态机状态
        val existingStatus = stateMachineService.getCurrentStateByOrderNo(orderNo)
        
        // 如果订单已在 LOCAL_INITIALIZED 或之后的状态，直接返回
        if (existingStatus != null && existingStatus != OrderStatus.INIT) {
            log.info("Order already processed, skipping: orderNo=$orderNo")
            return
        }
        
        // 订单存在但仍在 INIT 状态，重新处理
        stateMachineService.initializeStateMachineByOrderNo(orderNo, OrderStatus.LOCAL_INITIALIZED)
        return
    }
    
    // 创建新订单
    val order = Order.fromPrApproved(orderNo, ...)
    val savedOrder = orderRepository.save(order)
    
    // 初始化状态机
    stateMachineService.initializeStateMachineByOrderNo(orderNo, OrderStatus.LOCAL_INITIALIZED)
}

4. Fork/Join 并行等待详解

并行等待流程

flowchart TB A[PR 审批通过] --> B[PR_APPROVED 事件] B --> C[INIT → LOCAL_INITIALIZED] C --> D[保存 Order 到 DB] D --> E[发送 COE 到 Kafka] E --> F[LOCAL_INITIALIZED → FACTORY_ORDER_SUBMITTED] F --> G[同步 Deal 服务 REST API] G --> H[Fork: FACTORY_ORDER_SUBMITTED] H --> I[等待 VOM] H --> J[等待 DOM] I --> K{收到 VOM} K -->|VOM| L[FIRST_VOM_RECEIVED] K -->|VOM_FAILED| M[ORDER_INITIALIZE_FAILED] J --> N{收到 DOM} N -->|DOM| O[FIRST_DOM_RECEIVED] L --> P{已有 DOM?} O --> Q{已有 VOM?} P -->|是| R[Join → ORDER_INITIALIZE_SUCCEED] Q -->|是| R P -->|否| L Q -->|否| O

等待规则：

  • FACTORY_ORDER_SUBMITTED 是 Fork 状态，等待 VOM 和 DOM 事件
  • 先收到 VOM → 进入 FIRST_VOM_RECEIVED，等待 DOM
  • 先收到 DOM → 进入 FIRST_DOM_RECEIVED，等待 VOM
  • 两者都收到 → Join 成功，进入 ORDER_INITIALIZE_SUCCEED
  • 收到 VOM_FAILED → 直接进入 ORDER_INITIALIZE_FAILED

11. 最佳实践总结

1. 使用 orderNo 作为状态机 ID
外部系统友好，无需关注内部 orderId，调试直观。

2. 状态同步使用 REQUIRES_NEW 事务
独立事务确保状态同步不受外部事务影响，失败不影响状态机执行。

3. Action 中实现幂等检查
重复消息不会产生副作用，检查订单状态后再处理。

4. 所有 Kafka 消息携带 orderNo
orderNo 是自然主键，外部系统必须提供。

5. 状态机停止时同步状态
在 stateMachineStopped 回调中同步，避免每次状态变更时的性能开销。
