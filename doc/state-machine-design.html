订单生命周期状态机设计文档

1. 状态机总览

本设计使用 Barrier Aggregate 模式实现订单完整生命周期，包括订单初始化、PR Accept、CDOA Accept 三个阶段。状态机使用 orderNo 作为唯一标识，实现外部系统统一使用 orderNo 进行交互。

核心特性：
  • orderNo 绑定：状态机以 orderNo 作为 machineId，外部系统无需关注内部 orderId
  • Barrier Aggregate 模式：每个阶段需要等待多个屏障全部通过后才能进入下一阶段
  • 市场感知：不同市场需要不同的屏障配置
  • 失败重试：PR Accept 失败后支持重试
  • 线程安全：通过 OrderCommandService → TaskSpec → ShedLock 保证同一订单的事件串行执行
  • 事务最小化：事务仅包裹状态持久化，Action 在事务外执行
  • 精细化重试控制：区分 BusinessError（不重试）和 TechnicalError（重试）

5. 事务与重试架构

5.1 事务边界设计

OrderStateMachineTaskSpec.executeWithLock() [无事务]
  ├── 获取订单状态（只读）
  ├── StateMachine.sendEvent()
  │   └── Actions.execute() ← 无事务，允许耗时操作
  │       ├── HTTP 调用
  │       ├── Kafka 发送
  │       └── Barrier 初始化
  └── StateMachineListener.onStateChanged() [Transaction]
      ├── 更新 orders.status
      └── 保存 state_machine_history
设计优势：
  • 事务最小化：仅包裹 orders.status 和 state_machine_history 的原子写入
  • 避免长事务：Action 中的 HTTP、Kafka 操作不在事务中
  • 锁竞争减少：数据库锁持有时间最短

5.2 重试控制

场景 ActionResult StateChangeResult.failureReason 重试
分布式锁获取失败 - - ✅ LockingTaskSpec 返回 retryable=true
Event 不匹配 transition - INVALID_TRANSITION ❌
Action 参数校验失败 BusinessError BUSINESS_ERROR ❌
Action 数据库失败 TechnicalError TECHNICAL_ERROR ✅
Action HTTP/Kafka 失败 TechnicalError TECHNICAL_ERROR ✅
Listener 持久化失败 - 抛异常给 TaskSpec ✅

5.3 ActionResult 密封类

sealed class ActionResult {
    data class Success(val nextEvent: Enum<*>? = null) : ActionResult()
    
    sealed class Failure : ActionResult() {
        abstract val reason: String
        
        // 业务规则失败，不应重试（如：参数校验失败、订单已存在）
        data class BusinessError(override val reason: String) : Failure()
        
        // 技术故障，应该重试（如：数据库连接失败、HTTP 调用超时）
        data class TechnicalError(
            override val reason: String,
            val cause: Throwable? = null,
        ) : Failure()
    }
}

6. 线程安全架构

重要：Spring State Machine 不是线程安全的

多个线程同时操作同一个订单的状态机会导致状态混乱。本项目通过 TaskSpec + ShedLock 保证线程安全。

7. Barrier Aggregate 模式

7.1 单表 JSONB 设计

CREATE TABLE barrier_aggregate (
    id BIGSERIAL PRIMARY KEY,
    aggregate_type VARCHAR(500) NOT NULL,
    aggregate_key VARCHAR(255) NOT NULL,
    required_barriers JSONB NOT NULL,
    passed_barriers JSONB NOT NULL DEFAULT '[]',
    initialized_at TIMESTAMP NOT NULL,
    version BIGINT NOT NULL DEFAULT 0,
    CONSTRAINT uk_barrier_aggregate UNIQUE(aggregate_type, aggregate_key)
);

10. 测试注意事项

自动转换状态的验证

Spring State Machine 在处理事件时，如果存在多个连续的自动转换，STATE_CHANGED 回调的顺序可能与实际转换顺序相反。

测试策略：不直接验证中间状态，而是验证 Action 执行结果和最终状态。
