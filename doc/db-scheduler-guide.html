DB-Scheduler 使用指南

一、简介

DB-Scheduler 是一个基于数据库的持久化任务调度框架，通过乐观锁机制确保任务在分布式环境中的正确执行。

🎯 持久化存储

任务状态存储在数据库中

🔒 分布式安全

基于数据库锁机制

⏰ 灵活调度

支持一次性、周期性任务

🛠️ Spring 集成

提供 Spring Boot Starter

二、为什么需要 TaskSpec 封装？

状态机需要线程安全和重试控制

多个线程同时操作同一个订单的状态机会导致状态混乱。我们需要 TaskSpec + ShedLock 保证线程安全，同时通过 ActionResult 实现精细化重试控制。

需求原生 APITaskSpec 封装
分布式锁❌ 不支持✅ ShedLock 集成
同一订单串行执行❌ 需手动实现✅ lockKeyProvider
语义化 API❌ 需构造 TaskInstance✅ taskScheduler.submit()
精细化重试❌ 只有 fail/reschedule✅ BusinessError/TechnicalError

3.2 TaskResult 密封类

TaskResult.kt
sealed class TaskResult {
    data class Success(val message: String? = null) : TaskResult()
    
    data class Fail(
        val reason: String,
        val cause: Throwable? = null,
        val retryable: Boolean = true,  // 关键：控制是否重试
        val delay: Duration? = null,
    ) : TaskResult()
    
    companion object {
        fun success(message: String? = null): TaskResult = Success(message)
        fun fail(reason: String, retryable: Boolean = true): TaskResult = Fail(reason, retryable = retryable)
        fun failWithoutRetry(reason: String): TaskResult = Fail(reason, retryable = false)
    }
}

3.3 LockingTaskSpec（分布式锁）

LockingTaskSpec.kt
abstract class LockingTaskSpec
(
    private val lockProvider: LockProvider,
    private val lockKeyProvider: (TaskContext
) -> String,
) : TaskSpec
 {
    
    final override fun execute(context: TaskContext
): TaskResult {
        val lockKey = lockKeyProvider(context)
        val lock = lockProvider.lock(lockKey)
        
        if (lock.isEmpty) {
            // 锁获取失败 → 返回可重试
            return TaskResult.fail("Lock unavailable: $lockKey", retryable = true)
        }
        
        return try {
            executeWithLock(context)
        } finally {
            lock.get().unlock()
        }
    }
    
    protected abstract fun executeWithLock(context: TaskContext
): TaskResult
}

四、重试策略

4.1 重试场景

场景 retryable 重试次数
分布式锁获取失败 ✅ true 无限（直到成功）
Event 不匹配 transition ❌ false 0
Action 业务错误 ❌ false 0
Action 技术错误 ✅ true maxRetries
Listener 持久化失败 ✅ true maxRetries

4.2 RetryStrategy 配置

重试策略选项
sealed class RetryStrategy {
    // 固定延迟
    data class FixedDelay(val delay: Duration) : RetryStrategy()
    
    // 指数退避
    data class ExponentialBackoff(
        val initialDelay: Duration,
        val multiplier: Double = 2.0,
    ) : RetryStrategy()
    
    // 不重试
    object NoRetry : RetryStrategy()
}

五、OrderStateMachineTaskSpec 实现

OrderStateMachineTaskSpec.kt
@Component
class OrderStateMachineTaskSpec(
    private val orderJpaRepository: OrderJpaRepository,
    private val transitionTable: TransitionTable,
    private val stateMachineListener: StateChangedListener,
    lockProvider: LockProvider,
) : LockingTaskSpec(
    lockProvider = lockProvider,
    lockKeyProvider = { ctx -> "order:${ctx.payload.orderNo}" },
) {
    override val taskName: String = "order-state-machine"
    override val maxRetries: Int = 5
    override val retryStrategy: RetryStrategy =
        RetryStrategy.exponentialBackoff(Duration.ofSeconds(3), 2.0)
    
    override fun executeWithLock(context: TaskContext): TaskResult {
        val payload = context.payload
        
        // Step 1: 获取当前状态（只读）
        val order = orderJpaRepository.findByOrderNo(payload.orderNo)
        val currentState = order?.status ?: OrderStatus.INIT
        
        // Step 2: 执行状态机（无事务）
        val stateMachine = StateMachine.restore(
            id = payload.orderNo,
            currentState = currentState,
            initialState = OrderStatus.INIT,
            transitionTable = transitionTable,
            listener = stateMachineListener,  // Listener 内部使用事务
        )
        val result = stateMachine.sendEvent(payload.event, headers)
        
        // Step 3: 根据结果决定重试策略
        return when {
            result.accepted -> TaskResult.success()
            result.shouldRetry() -> TaskResult.fail("Technical error", retryable = true)
            else -> TaskResult.failWithoutRetry("Rejected: ${result.failureReason}")
        }
    }
}

7.1 查看 task 状态

SQL 查询
-- 查看所有待处理任务
SELECT task_name, task_instance, consecutive_failures, 
       scheduled_time, last_failure
FROM scheduled_tasks
ORDER BY scheduled_time;

-- 查看失败次数最多的任务
SELECT task_name, task_instance, consecutive_failures
FROM scheduled_tasks
WHERE consecutive_failures > 0
ORDER BY consecutive_failures DESC;

7.2 日志关键字

日志关键字 含义
Lock unavailable 分布式锁获取失败，会自动重试
Task {} failed (retryable) 技术错误，会重试
Task {} failed (no retry) 业务错误，不会重试
Task {} exceeded max retries 达到最大重试次数，任务被移除
