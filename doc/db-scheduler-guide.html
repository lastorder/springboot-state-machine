DB-Scheduler 使用指南

一、简介

DB-Scheduler 是一个基于数据库的持久化任务调度框架，由瑞典工程师 Gustav Kärrlsson 开发。 它将任务调度信息存储在数据库中，通过乐观锁机制确保任务在分布式环境中的正确执行， 是 Spring Boot 应用中实现定时任务和延迟任务的理想选择。

🎯 持久化存储

任务状态存储在数据库中，应用重启后任务不丢失

🔒 分布式安全

基于数据库锁机制，天然支持集群部署，无重复执行

⏰ 灵活调度

支持一次性任务、周期性任务、延迟任务、Cron 表达式

🛠️ Spring 集成

提供 Spring Boot Starter，开箱即用

二、解决的问题

2.1 为什么需要 DB-Scheduler？

在传统的任务调度方案中，我们面临以下挑战：

方案 问题 DB-Scheduler 解决
Timer / ScheduledExecutorService 内存级，重启丢失；无持久化 数据库持久化，重启后继续执行
Spring @Scheduled 集群环境下重复执行；无法动态调度 分布式锁机制，支持动态任务
Quartz 重量级；配置复杂；数据库表多 轻量级；单表存储；零配置启动
Redis 延迟队列 依赖 Redis；需要额外组件 仅依赖数据库；无额外中间件

2.2 本项目的具体问题

问题背景：Spring State Machine 不是线程安全的，多个线程同时操作同一个订单的状态机会导致状态混乱。 我们需要一种机制确保同一个订单的事件按顺序执行。

核心冲突场景

并发场景： ┌─────────┐ ┌─────────┐ ┌─────────┐ │ 线程 A │────▶│ 线程 B │────▶│ 状态机 │ ❌ 状态混乱！ │ PAY │ │ CANCEL │ │ (并发) │ └─────────┘ └─────────┘ └─────────┘ 串行化方案： ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ │ HTTP/ │────▶│ Command │────▶│ DB- │────▶│ 状态机 │ │ Kafka │ │ Inbox │ │Scheduler│ │ (串行) │ └─────────┘ └─────────┘ └─────────┘ └─────────┘ (持久化) (One Task ✅ 安全！ Per Order)

DB-Scheduler 的解决方案

通过 One Task Per Order 模式，确保每个订单只有一个活跃任务，所有事件排队执行：

  1. 任务唯一性：使用 task_id = orderId 作为任务标识，DB-Scheduler 使用 SELECT FOR UPDATE 确保同一时刻只有一个线程执行该任务
  2. 串行执行：任务从 Command Inbox 取出下一个待执行命令，执行完毕后再取下一个
  3. 自我重调度：任务执行完一个命令后，检查是否还有待执行命令，有的话重新调度自己

3.2 任务类型

类型 说明 使用场景
OneTimeTask 一次性任务，执行一次后自动删除 订单状态机任务、超时检查
RecurringTask 周期性任务，按固定间隔执行 清理过期命令

3.3 数据库表结构

DB-Scheduler 只需要一张表：

schema.sql
CREATE TABLE scheduled_tasks (
    task_name           VARCHAR(100) NOT NULL,
    task_instance       VARCHAR(100) NOT NULL,
    task_data           BYTEA,
    execution_time      TIMESTAMP WITH TIME ZONE NOT NULL,
    picked              BOOLEAN NOT NULL DEFAULT FALSE,
    picked_by           VARCHAR(50),
    last_success        TIMESTAMP WITH TIME ZONE,
    last_failure        TIMESTAMP WITH TIME ZONE,
    consecutive_failures INT,
    last_heartbeat      TIMESTAMP WITH TIME ZONE,
    version             BIGINT NOT NULL,
    PRIMARY KEY (task_name, task_instance)
);

关键设计： 主键是 (task_name, task_instance) 组合，这意味着同一个任务名称下，同一个 instance 只能存在一条记录。 这正是我们实现 "One Task Per Order" 的基础！

4.3 定义 TaskData

任务数据必须实现 Serializable：

TaskData.kt
data class OrderTaskData(
    val orderId: Long,
) : Serializable {
    companion object {
        private const val serialVersionUID: Long = 1L

4.4 创建 OneTimeTask

OrderStateMachineTaskConfig.kt
@Configuration
class OrderStateMachineTaskConfig(
    private val commandInboxService: CommandInboxService,
    private val commandInboxRepository: CommandInboxRepository,
    private val stateMachineService: StateMachineService,
) {
    companion object {
        const val TASK_NAME = "order-state-machine"
    }

    @Bean
    fun orderStateMachineTask(): OneTimeTask<OrderTaskData> {
        return object : OneTimeTask<OrderTaskData>(
            TASK_NAME,
            OrderTaskData::class.java,
            FailureHandler.OnFailureRetryLater(Duration.ofMinutes(1)),
        ) {
            override fun executeOnce(
                taskInstance: TaskInstance<OrderTaskData>,
                executionContext: ExecutionContext,
            ) {
                val orderId = taskInstance.data.orderId
                
                // 1. 取出下一个待执行命令
                val command = commandInboxRepository.findNextPendingCommand(orderId)
                    ?: return  // 没有待执行命令，任务结束
                
                // 2. 执行命令
                processCommand(command, orderId)
                
                // 3. 自我重调度（如果还有待执行命令）
                scheduleSelfIfHasPendingCommands(orderId)
            }
        }
    }
}

4.5 创建 RecurringTask

ValidationTimeoutTaskConfig.kt
@Bean
fun commandCleanupTask(): RecurringTask<Void> =
    Tasks
        .recurring("command-cleanup", Schedules.fixedDelay(Duration.ofHours(1)))
        .execute { _: TaskInstance<Void>, _: ExecutionContext ->
            log.info("Running command cleanup task")
            
            // 清理过期命令
            val expiredCommands = commandInboxRepository.findExpiredCommands(Instant.now())
            expiredCommands.forEach { command ->
                commandInboxRepository.save(command.copy(status = CommandStatus.EXPIRED))
            }
            
            // 删除30天前已完成的命令
            val cutoffDate = Instant.now().minusSeconds(30 * 24 * 60 * 60L)
            commandInboxRepository.deleteByStatusAndProcessedAtBefore(
                CommandStatus.COMPLETED,
                cutoffDate,
            )
        }

4.6 手动调度任务

CommandInboxService.kt
@Service
class CommandInboxService(
    private val schedulerClient: SchedulerClient,
) {
    fun scheduleOrderTask(orderId: Long) {
        val taskInstance = TaskInstance<OrderTaskData>(
            TASK_NAME,
            orderId.toString(),  // task_instance = orderId，确保唯一性
            OrderTaskData(orderId),
        )
        
        // 立即调度，如果已存在则忽略
        schedulerClient.scheduleIfNotExists(taskInstance, Instant.now())
    }
}

schedule() vs scheduleIfNotExists()

schedule() 会检查是否已存在相同任务，已存在则抛出异常。
scheduleIfNotExists() 会检查是否已存在，已存在则静默忽略。

在 "One Task Per Order" 模式下，推荐使用 scheduleIfNotExists()。

4.7 调度策略

方法 说明 示例
schedule(task, time) 在指定时间执行 schedule(task, Instant.now().plusSeconds(60))
scheduleIfNotExists(task, time) 如果不存在则调度 scheduleIfNotExists(task, Instant.now())
reschedule(task, time) 重新调度（更新执行时间） reschedule(task, Instant.now().plusSeconds(30))
cancel(task) 取消任务 cancel(taskInstance)

4.8 失败处理

FailureHandler.OnFailureRetryLater(Duration.ofMinutes(1))  // 失败后1分钟重试
FailureHandler.MaxRetries(3)  // 最多重试3次
FailureHandler.OnFailureRetryLater(Duration.ofMinutes(1),  // 求策略
    FailureHandler.MaxRetries(5)
)

5.2 关键实现细节

命令优先级排序

CommandInboxRepository.kt
-- 按 priority DESC, id ASC 排序，确保高优先级先执行
SELECT * FROM command_inbox 
WHERE order_id = :orderId 
  AND status = 'PENDING'
ORDER BY priority DESC, id ASC
LIMIT 1
优先级 值 事件类型
URGENT 200 CANCEL, REFUND, PAYMENT 操作
HIGH 100 Validation, Confirmation 操作
NORMAL 0 标准操作

自我重调度机制

OrderStateMachineTaskConfig.kt
private fun scheduleSelfIfHasPendingCommands(orderId: Long) {
    Thread.sleep(NEXT_COMMAND_DELAY_MS)  // 100ms 延迟，避免忙等待
    
    val pendingCount = commandInboxRepository.countPendingCommands(orderId)
    
    if (pendingCount > 0) {
        val taskInstance = TaskInstance(TASK_NAME, orderId.toString(), OrderTaskData(orderId))
        schedulerClient.scheduleIfNotExists(taskInstance, Instant.now())
        log.debug("Rescheduled task: orderId={}, pendingCount={}", orderId, pendingCount)
    }
}

为什么需要 100ms 延迟？

如果新命令刚写入但事务还未提交，立即查询可能看不到。100ms 延迟给事务提交留出时间。

创建订单时，调度一个超时检查任务：

fun scheduleValidationTimeout(orderId: Long) {

任务执行时检查订单是否仍在验证中：

ValidationTimeoutTaskConfig.kt
override fun executeOnce(
    taskInstance: TaskInstance<TimeoutCheckData>,
    executionContext: ExecutionContext,
) {
    val orderId = taskInstance.data.orderId
    val order = orderRepository.findById(orderId)
    
    // 只有仍在 PENDING_VALIDATION 才触发超时
    if (order != null && order.status == OrderStatus.PENDING_VALIDATION) {
        commandInboxService.submitCommand(
            orderId = orderId,
            event = OrderEvent.VALIDATION_TIMEOUT,
            source = CommandSource.SCHEDULED,
            priority = CommandPriority.HIGH,
        )
    }
}

六、最佳实践

6.1 任务粒度设计

粒度越小越好

每个任务只做一件事。如果需要多步骤，使用 Command Inbox 拆分成多个命令。

本项目中，一个任务处理一个订单的多个命令，而不是一个任务处理多个订单。

6.2 幂等性设计

任务执行必须是幂等的，因为网络抖动可能导致任务重复执行。

本项目使用 Command Inbox 的状态字段（PENDING → COMPLETED/SKIPPED）确保幂等性。

6.3 心跳检测

DB-Scheduler 会定期更新 last_heartbeat 字段，如果检测到心跳超时，会认为任务已死锁，重新调度。

application.yml
db-scheduler:
  heartbeat-interval: 1m  # 心跳间隔
  # 如果 last_heartbeat 超过 2 * heartbeat-interval 未更新，认为任务已死

6.4 监控与告警

建议监控以下指标：

  • 任务执行延迟：execution_time 与当前时间的差距
  • 任务失败次数：consecutive_failures
  • 死锁任务数量：长时间未更新心跳的任务
  • 待执行任务积压：WHERE picked = false 的记录数

6.5 性能优化

配置项 建议值 说明
threads CPU 核心数 执行线程数，过多会增加锁竞争
polling-interval 5s ~ 10s 扫描间隔，过短增加数据库压力
immediate-execution-enabled true 新任务立即执行，无需等待扫描周期

6.6 与其他方案对比

graph TD subgraph "DB-Scheduler" A1[简单易用] --> B1[适合中小规模] A2[数据库持久化] --> B2[无额外依赖] A3[分布式安全] --> B3[集群部署友好] end subgraph "Quartz" C1[功能完善] --> D1[适合大规模] C2[集群支持] --> D2[配置复杂] C3[数据库持久化] --> D3[多表管理] end subgraph "Redis 延迟队列" E1[高性能] --> F1[适合高并发] E2[内存存储] --> F2[依赖 Redis] E3[分布式安全] --> F3[Redis 集群] end

选择建议：

DB-Scheduler：中小规模、需要持久化、不想引入额外中间件
Quartz：大规模、需要复杂调度策略
Redis：高性能场景、已有 Redis 基础设施

七、总结

DB-Scheduler 是一个轻量、可靠的持久化任务调度框架，特别适合需要分布式任务协调的 Spring Boot 应用。 在本项目中，它通过 One Task Per Order 模式解决了 Spring State Machine 的线程安全问题， 确保了订单状态变更的可靠性和一致性。

核心设计要点：

  1. 任务唯一性：使用 orderId 作为 task_instance，确保每个订单只有一个活跃任务
  2. 串行执行：Task 循环处理命令，保证顺序性
  3. 自我重调度：处理完一个命令后检查是否有新命令，有则重新调度
  4. 失败处理：命令失败标记为 SKIPPED，不影响后续命令执行
