DB-Scheduler 使用指南

一、简介

DB-Scheduler 是一个基于数据库的持久化任务调度框架，由瑞典工程师 Gustav Kärrlsson 开发。 它将任务调度信息存储在数据库中，通过乐观锁机制确保任务在分布式环境中的正确执行， 是 Spring Boot 应用中实现定时任务和延迟任务的理想选择。

🎯 持久化存储

任务状态存储在数据库中，应用重启后任务不丢失

🔒 分布式安全

基于数据库锁机制，天然支持集群部署，无重复执行

⏰ 灵活调度

支持一次性任务、周期性任务、延迟任务、Cron 表达式

🛠️ Spring 集成

提供 Spring Boot Starter，开箱即用

2.2 数据库表结构

DB-Scheduler 只需要一张表：

schema.sql
CREATE TABLE scheduled_tasks (
    task_name           VARCHAR(100) NOT NULL,
    task_instance       VARCHAR(100) NOT NULL,
    task_data           BYTEA,
    execution_time      TIMESTAMP WITH TIME ZONE NOT NULL,
    picked              BOOLEAN NOT NULL DEFAULT FALSE,
    picked_by           VARCHAR(50),
    last_success        TIMESTAMP WITH TIME ZONE,
    last_failure        TIMESTAMP WITH TIME ZONE,
    consecutive_failures INT,
    last_heartbeat      TIMESTAMP WITH TIME ZONE,
    version             BIGINT NOT NULL,
    PRIMARY KEY (task_name, task_instance)
);

关键设计： 主键是 (task_name, task_instance) 组合，这意味着同一个任务名称下，同一个 instance 只能存在一条记录。

2.5 调度任务

OrderService.kt
@Service
class OrderService(
    private val schedulerClient: SchedulerClient,
) {
    fun scheduleOrderEvent(orderId: Long, event: String) {
        val taskInstance = TaskInstance(
            OrderTaskConfig.TASK_NAME,
            "order-$orderId-$event",  // instanceId 必须唯一
            OrderTaskData(orderId, event),
        )
        
        // 方式1：调度任务（已废弃）
        schedulerClient.schedule(taskInstance, Instant.now())
        
        // 方式2：如果不存在则调度（推荐）
        val scheduled = schedulerClient.scheduleIfNotExists(taskInstance, Instant.now())
        if (!scheduled) {
            log.warn("Task already exists: orderId=$orderId, event=$event")
        }
    }
}

三、原生 API 的问题

3.1 本项目的具体需求

问题背景：Spring State Machine 不是线程安全的，多个线程同时操作同一个订单的状态机会导致状态混乱。 我们需要一种机制确保同一个订单的事件按顺序执行。

核心冲突场景

并发场景：
┌─────────┐     ┌─────────┐     ┌─────────┐
│ 线程 A   │────▶│ 线程 B   │────▶│ 状态机   │  ❌ 状态混乱！
│ VOM     │     │ DOM     │     │ (并发)   │
└─────────┘     └─────────┘     └─────────┘

串行化方案：
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ Kafka   │────▶│ Task    │────▶│ DB-     │────▶│ 状态机   │
│ Event   │     │Scheduler│     │Scheduler│     │ (串行)   │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
                (TaskSpec)      (One Task      ✅ 安全！
                                 Per Order)

3.2 原生 API 的痛点

1. 缺乏分布式锁
原生 API 只保证同一 task_name + task_instance 不会重复调度，但无法保证同一订单的不同事件串行执行。

例如：order-123-VOM 和 order-123-DOM 是两个不同的任务，可能并发执行。

2. 失败处理不灵活
原生 API 通过 FailureHandler 配置重试策略，但无法自定义复杂的失败逻辑。

例如：需要区分"业务失败"（不重试）和"技术失败"（重试）。

3. API 不够语义化
每次调度都需要构造 TaskInstance，代码冗余。

原生：

val taskInstance = TaskInstance(TASK_NAME, instanceId, data)
schedulerClient.scheduleIfNotExists(taskInstance, Instant.now())

4. 任务定义分散
任务定义（OneTimeTask）和任务调度（SchedulerClient）分离，难以统一管理。

3.3 对比表

需求 原生 API TaskSpec 封装
分布式锁 ❌ 不支持 ✅ ShedLock 集成
业务失败 vs 技术失败 ❌ 无法区分 ✅ TaskResult 密封类
语义化 API ❌ 需构造 TaskInstance ✅ taskScheduler.submit(spec, id, payload)
统一管理 ❌ 分散定义 ✅ 自动扫描注册
锁粒度控制 ❌ 不支持 ✅ lockKeyProvider 自定义

TaskSpec.kt
interface TaskSpec
 {
    val taskName: String
    val maxRetries: Int
        get() = 3
    val payloadClass: Class


    fun execute(context: TaskContext
): TaskResult

    fun onMaxRetriesExceeded(context: TaskContext

4.3 TaskResult 密封类

区分不同的执行结果：

TaskResult.kt
sealed class TaskResult {
    // 成功：任务完成
    data object Success : TaskResult()
    
    // 重试：技术失败，稍后重试
    data class Retry(val delay: Duration = Duration.ofMinutes(1)) : TaskResult()
    
    // 失败：业务失败，不重试
    data class Fail(val reason: String? = null) : TaskResult()

    companion object {
        fun success(): TaskResult = Success
        fun retry(delay: Duration = Duration.ofMinutes(1)): TaskResult = Retry(delay)
        fun fail(reason: String? = null): TaskResult = Fail(reason)
    }
}

关键设计：

Success - 任务成功完成
Retry - 技术（可重试）失败，如网络超时、数据库连接失败
Fail - 业务（不可重试）失败，如状态机拒绝事件

4.4 LockingTaskSpec（分布式锁）

集成 ShedLock 实现分布式锁：

LockingTaskSpec.kt
abstract class LockingTaskSpec
(
    override val taskName: String,
    override val maxRetries: Int = 3,
    private val lockProvider: LockProvider,
    private val lockKeyProvider: (TaskContext
) -> String,
) : TaskSpec
 {

    override fun execute(context: TaskContext
): TaskResult {
        val lockKey = lockKeyProvider(context)
        val lock = lockProvider.lock(lockKey)
        
        return if (lock != null) {
            try {
                executeWithLock(context)
            } finally {
                lock.unlock()
            }
        } else {
            // 获取锁失败，稍后重试
            TaskResult.retry()
        }
    }

    abstract fun executeWithLock(context: TaskContext
): TaskResult
}

锁粒度控制：

通过 lockKeyProvider 自定义锁的粒度。

例如：{ ctx -> "order:${ctx.payload.orderId}" }
表示同一个 orderId 的所有事件会串行执行。

五、项目实战

5.1 OrderStateMachineTaskSpec 实现

OrderStateMachineTaskSpec.kt
@Component
class OrderStateMachineTaskSpec(
    private val stateMachineService: StateMachineService,
    lockProvider: LockProvider,
) : LockingTaskSpec(
    taskName = "order-state-machine",
    maxRetries = 5,
    lockProvider = lockProvider,
    lockKeyProvider = { ctx -> "order:${ctx.payload.orderId}" },
) {
    override fun executeWithLock(context: TaskContext): TaskResult {
        val orderId = context.payload.orderId
        val event = context.payload.event

        val success = stateMachineService.sendEvent(orderId, event)
        
        return if (success) {
            TaskResult.success()
        } else {
            // 业务失败：状态机拒绝事件，不重试
            TaskResult.fail("State machine rejected event: $event")
        }
    }
}

OrderCommandService.kt
@Service
class OrderCommandService(

六、最佳实践

6.1 任务粒度设计

粒度越小越好

每个任务只做一件事。本项目中，一个任务处理一个订单的一个事件。

6.2 幂等性设计

任务执行必须是幂等的，因为网络抖动可能导致任务重复执行。

使用 scheduleIfNotExists() 避免重复调度。

6.3 事务处理策略

TaskSpec execute() 方法不加 @Transactional

原因：
• db-scheduler 内部管理任务生命周期，需要控制事务边界
• 任务执行失败后需要更新 consecutive_failures，不能在业务事务内
• 分布式锁（ShedLock）与数据库事务混用可能导致死锁

场景 事务策略 实现方式
TaskSpec.execute() 无事务 调用带事务的 Service 方法
StateMachineService.sendEventByOrderNo() 无事务 Reactive 流不能与传统事务混用
OrderStatusSyncService.syncOrderStatusByOrderNo() @Transactional(REQUIRES_NEW) 独立事务，确保状态同步隔离
OrderStateMachineTaskSpec.kt - 事务处理
@Component
class OrderStateMachineTaskSpec(
    private val stateMachineService: StateMachineService,
    lockProvider: LockProvider,
) : LockingTaskSpec<OrderEventPayload>(
    taskName = "order-state-machine",
    maxRetries = 5,
    lockProvider = lockProvider,
    lockKeyProvider = { ctx -> "order:${ctx.payload.orderNo}" },  // 使用 orderNo 作为锁 key
) {
    // execute() 已在父类实现，提供了分布式锁
    // executeWithLock() 不加 @Transactional
    override fun executeWithLock(context: TaskContext<OrderEventPayload>): TaskResult {
        val orderNo = context.payload.orderNo
        val event = context.payload.event
        
        // StateMachineService.sendEventByOrderNo() 不加事务
        // 内部的 OrderStatusSyncService 使用 REQUIRES_NEW 独立事务
        val success = stateMachineService.sendEventByOrderNo(orderNo, event)
        
        return if (success) {
            TaskResult.success()
        } else {
            // 业务失败：状态机拒绝事件，不重试
            TaskResult.fail(retryable = false, reason = "State machine rejected event")
        }
    }
}

事务边界设计原则

1. TaskSpec 不加事务：让 db-scheduler 管理任务状态
2. 调用带事务的 Service：业务操作在 Service 层处理事务
3. 状态同步用 REQUIRES_NEW：确保状态同步失败不影响任务状态
4. 分布式锁在事务外：ShedLock 的 lock/unlock 在事务边界外

6.4 锁粒度策略

锁 Key 使用 orderNo 而非 orderId

本项目已将状态机绑定改为使用 orderNo 作为 machineId， 因此锁 Key 也应使用 orderNo。

锁粒度 lockKeyProvider 适用场景
订单级（推荐） { ctx -> "order:${ctx.payload.orderNo}" } 同一订单的事件串行执行
用户级 { ctx -> "user:${ctx.payload.userId}" } 同一用户的操作串行执行
全局级 { ctx -> "global" } 所有任务串行执行（不推荐）

6.5 失败处理策略

失败类型 返回值 行为
技术失败（网络、DB） TaskResult.Fail(retryable=true) 重试，最多 maxRetries 次
业务失败（状态机拒绝） TaskResult.Fail(retryable=false, reason) 不重试，调用 onFinalFailure()
成功 TaskResult.Success 任务完成

6.6 监控指标

  • 任务执行延迟：execution_time 与当前时间的差距
  • 任务失败次数：consecutive_failures
  • 死锁任务数量：长时间未更新心跳的任务
  • 待执行任务积压：WHERE picked = false 的记录数

七、总结

DB-Scheduler 是一个轻量、可靠的持久化任务调度框架。原生 API 适合简单场景，但在复杂业务中存在以下问题：

  • 缺乏分布式锁
  • 无法区分业务失败和技术失败
  • API 不够语义化

通过 TaskSpec 封装模式，我们解决了这些问题：

  1. 分布式锁：ShedLock 确保同一订单的事件串行执行
  2. TaskResult 密封类：区分成功、重试、失败三种状态
  3. 语义化 API：taskScheduler.submit(spec, id, payload)
  4. 自动配置：TaskSpecAutoConfiguration 扫描并注册所有 TaskSpec beans
  5. 锁粒度控制：lockKeyProvider 自定义锁粒度
