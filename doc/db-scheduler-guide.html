DB-Scheduler 使用指南

一、简介

DB-Scheduler 是一个基于数据库的持久化任务调度框架，通过乐观锁机制确保任务在分布式环境中的正确执行。

🎯 持久化存储

任务状态存储在数据库中

🔒 分布式安全

基于数据库锁机制

⏰ 灵活调度

支持一次性、周期性任务

🛠️ Spring 集成

提供 Spring Boot Starter

二、为什么需要 TaskSpec 封装？

Spring State Machine 不是线程安全的

多个线程同时操作同一个订单的状态机会导致状态混乱。我们需要 TaskSpec + ShedLock 保证线程安全。

需求原生 APITaskSpec 封装
分布式锁❌ 不支持✅ ShedLock 集成
同一订单串行执行❌ 需手动实现✅ lockKeyProvider
语义化 API❌ 需构造 TaskInstance✅ taskScheduler.submit()

sealed class TaskResult {
    data object Success : TaskResult()
    data class Retry(val delay: Duration = Duration.ofMinutes(1)) : TaskResult()
    data class Fail(val reason: String? = null) : TaskResult()
}

3.3 LockingTaskSpec（分布式锁）

LockingTaskSpec.kt
abstract class LockingTaskSpec
(
    override val taskName: String,
    private val lockProvider: LockProvider,
    private val lockKeyProvider: (TaskContext
) -> String,
) : TaskSpec
 {
    override fun execute(context: TaskContext
): TaskResult {
        val lockKey = lockKeyProvider(context)
        val lock = lockProvider.lock(lockKey)
        return if (lock != null) {
            try { executeWithLock(context) } finally { lock.unlock() }
        } else { TaskResult.retry() }
    }
    abstract fun executeWithLock(context: TaskContext
): TaskResult
}

锁粒度控制：
{ ctx -> "order:${ctx.payload.orderNo}" } 表示同一个 orderNo 的所有事件会串行执行。

五、事务处理

TaskSpec execute() 不加 @Transactional

使用 TransactionTemplate 手动控制事务边界，避免 Spring AOP 问题和分布式锁死锁。

六、最佳实践

1. 任务粒度越小越好 - 一个任务处理一个订单的一个事件

2. 幂等性设计 - 使用 scheduleIfNotExists()

3. 锁 Key 使用 orderNo - 与状态机 machineId 保持一致

4. 区分失败类型 - 业务失败返回 Fail，技术失败返回 Retry
