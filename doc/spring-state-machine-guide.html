Spring State Machine 使用指南

一、简介

Spring State Machine 是 Spring 官方提供的状态机框架， 基于有限状态机（Finite State Machine）理论，帮助开发者在业务系统中实现复杂的状态流转逻辑。 它提供了声明式的配置方式、丰富的状态转换机制、以及与 Spring 生态的深度集成。

🎯 声明式配置

通过 Builder API 或注解配置状态和转换，代码清晰易维护

🔄 丰富的转换模式

支持外部转换、内部转换、Fork/Join、Entry/Exit Action 等

💾 状态持久化

提供 JPA、Redis、MongoDB 等多种持久化方案

🔐 Guard 条件

转换前可配置守卫条件，实现复杂的业务校验

📢 事件驱动

事件触发的状态转换，与消息队列天然契合

🧪 测试支持

提供 StateMachineTestPlan 简化测试

二、解决的问题

2.1 为什么需要状态机？

在业务系统中，状态管理是一个常见但容易出错的问题：

传统的方式（if-else / switch）：

if (status == "INIT") {
    if (event == "PR_APPROVED") {
        status = "LOCAL_INITIALIZED"
    }
} else if (status == "LOCAL_INITIALIZED") {
    if (event == "VOM") {
        // ❌ 状态混乱！还在等待 COE 发送
    }
} ... // 😱 难以维护！

问题：
1. 状态转换逻辑分散，没有统一入口
2. 非法转换难以防止
3. 状态图谱不直观
4. 测试困难

Spring State Machine 提供了更优雅的解决方案：

  • 集中配置：所有状态和转换在一处定义，清晰可见
  • 类型安全：使用枚举定义状态和事件，编译期检查
  • 自动校验：非法转换会被自动拒绝
  • 可视化：状态图谱一目了然

2.2 本项目的业务场景

本项目是一个订单初始化系统，需要处理 VOM/DOM 并行等待：

核心需求

1. PR 审批通过后开始 本地初始化，保存订单到 DB
2. 发送 COE 到 Kafka
3. 同步订单状态到 Deal 服务
4. 并行等待 VOM 和 DOM 事件
5. VOM_FAILED 直接导致 初始化失败

三、核心概念

3.1 基础组件

graph LR A[State] --> B[StateMachine] C[Event] --> B B --> D[Transition] D --> E[Action] D --> F[Guard] subgraph "状态机核心" B end
组件 说明 项目中的定义
State 状态，枚举类型 OrderStatus：INIT, LOCAL_INITIALIZED, FACTORY_ORDER_SUBMITTED...
Event 事件，触发状态转换 OrderEvent：PR_APPROVED, VOM, DOM, VOM_FAILED
Transition 转换，定义 State → State 的路径 source=INIT, target=LOCAL_INITIALIZED, event=PR_APPROVED
Action 动作，转换时执行的逻辑 PrApprovedAction：保存订单到 DB
StateMachine 状态机实例 每个订单一个实例，通过 orderId 标识

3.2 状态分类

类型 说明 项目中的使用
Initial State 初始状态，状态机启动时进入 INIT
End State 终止状态，状态机停止 ORDER_INITIALIZE_SUCCEED, ORDER_INITIALIZE_FAILED
Fork State 分支状态，进入多个并行区域 FACTORY_ORDER_SUBMITTED
Join State 汇聚状态，等待所有并行区域完成 ORDER_INITIALIZE_SUCCEED

4.3 配置状态机

StateMachineConfig.kt
@Configuration
@EnableStateMachineFactory
class StateMachineConfig(
    private val prApprovedAction: PrApprovedAction,
    private val sendCoeAction: SendCoeAction,
    private val syncDealAction: SyncDealAction,
) : StateMachineConfigurerAdapter() {
    
    override fun configure(states: StateMachineStateConfigurer) {
        states
            .withStates()
            .initial(OrderStatus.INIT)
            .fork(OrderStatus.FACTORY_ORDER_SUBMITTED)
            .join(OrderStatus.ORDER_INITIALIZE_SUCCEED)
            .state(OrderStatus.LOCAL_INITIALIZED)
            .state(OrderStatus.FACTORY_ORDER_SUBMITTED, syncDealAction, null)
            .state(OrderStatus.FIRST_VOM_RECEIVED)
            .state(OrderStatus.FIRST_DOM_RECEIVED)
            .end(OrderStatus.ORDER_INITIALIZE_SUCCEED)
            .end(OrderStatus.ORDER_INITIALIZE_FAILED)
    }
    
    override fun configure(transitions: StateMachineTransitionConfigurer) {
        transitions
            // INIT → LOCAL_INITIALIZED
            .withExternal()
            .source(OrderStatus.INIT)
            .target(OrderStatus.LOCAL_INITIALIZED)
            .event(OrderEvent.PR_APPROVED)
            .action(prApprovedAction)
            .and()
            // LOCAL_INITIALIZED → FACTORY_ORDER_SUBMITTED
            .withExternal()
            .source(OrderStatus.LOCAL_INITIALIZED)
            .target(OrderStatus.FACTORY_ORDER_SUBMITTED)
            .action(sendCoeAction)
            .and()
            // Fork: FACTORY_ORDER_SUBMITTED → FIRST_VOM_RECEIVED + FIRST_DOM_RECEIVED
            .withFork()
            .source(OrderStatus.FACTORY_ORDER_SUBMITTED)
            .target(OrderStatus.FIRST_VOM_RECEIVED)
            .target(OrderStatus.FIRST_DOM_RECEIVED)
            .and()
            // Join: FIRST_VOM_RECEIVED + FIRST_DOM_RECEIVED → ORDER_INITIALIZE_SUCCEED
            .withJoin()
            .source(OrderStatus.FIRST_VOM_RECEIVED)
            .source(OrderStatus.FIRST_DOM_RECEIVED)
            .target(OrderStatus.ORDER_INITIALIZE_SUCCEED)
            .and()
            // VOM_FAILED → ORDER_INITIALIZE_FAILED
            .withExternal()
            .source(OrderStatus.FACTORY_ORDER_SUBMITTED)
            .target(OrderStatus.ORDER_INITIALIZE_FAILED)
            .event(OrderEvent.VOM_FAILED)
    }
}

4.4 Action 编写

PrApprovedAction.kt
@Component
class PrApprovedAction(
    private val orderService: OrderService,
) : Action {
    
    override fun execute(context: StateContext) {
        val orderId = context.message.headers["orderId", Long::class.java] ?: return
        
        // 从 Kafka 消息中获取订单信息并保存到 DB
        orderService.createOrderFromPrApproved(orderId, context.messagePayload)
    }
}

五、Fork/Join 并行模式

5.1 什么是 Fork/Join？

Fork/Join 是一种并行处理模式：Fork 将流程拆分为多个并行分支， Join 等待所有分支完成后汇聚。

stateDiagram-v2 direction TB [*] --> INIT INIT --> LOCAL_INITIALIZED: PR_APPROVED state FACTORY_ORDER_SUBMITTED { state fork_state <<fork>> state join_state <<join>> [*] --> fork_state fork_state --> FIRST_VOM_RECEIVED fork_state --> FIRST_DOM_RECEIVED FIRST_VOM_RECEIVED --> join_state: DOM FIRST_DOM_RECEIVED --> join_state: VOM join_state --> [*] } LOCAL_INITIALIZED --> FACTORY_ORDER_SUBMITTED: 发送 COE FACTORY_ORDER_SUBMITTED --> ORDER_INITIALIZE_SUCCEED: Join success FACTORY_ORDER_SUBMITTED --> ORDER_INITIALIZE_FAILED: VOM_FAILED

5.2 Fork/Join 流程说明

流程： 1. INIT ──PR_APPROVED──▶ LOCAL_INITIALIZED（保存订单） 2. LOCAL_INITIALIZED ──发送 COE──▶ FACTORY_ORDER_SUBMITTED 3. FACTORY_ORDER_SUBMITTED ──Fork──▶ FIRST_VOM_RECEIVED + FIRST_DOM_RECEIVED（并行） 4. FIRST_VOM_RECEIVED ──收到 DOM──▶ Join 5. FIRST_DOM_RECEIVED ──收到 VOM──▶ Join 6. Join 成功 ──────────────────────▶ ORDER_INITIALIZE_SUCCEED 异常分支： - 任意状态收到 VOM_FAILED ──▶ ORDER_INITIALIZE_FAILED

5.3 Join 条件

Join 的触发条件

Spring State Machine 的 Join 需要所有并行区域都到达自己的结束状态。

本项目中：
• FIRST_VOM_RECEIVED 需要收到 DOM 事件才能到达 Join
• FIRST_DOM_RECEIVED 需要收到 VOM 事件才能到达 Join
• 两者都到达后，自动触发 Join，进入 ORDER_INITIALIZE_SUCCEED

六、状态持久化

6.1 为什么需要持久化？

Spring State Machine 默认将状态保存在内存中。但在实际业务中：

  • 应用重启后状态丢失
  • 集群环境下状态不同步
  • 需要查询历史状态

6.2 配置持久化

application.yml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/orders
    driver-class-name: org.postgresql.Driver
  jpa:
    hibernate:
      ddl-auto: update

七、线程安全问题

7.1 问题分析

Spring State Machine 不是线程安全的！

官方文档明确指出：一个 StateMachine 实例不应该被多个线程同时访问。

原因：
• 状态机内部使用非线程安全的数据结构
• 状态变更过程中存在竞态条件
• 并发操作可能导致状态不一致

7.2 解决方案：TaskSpec + DB-Scheduler

本项目采用 TaskSpec 模式 解决线程安全问题：

graph LR A[Kafka Event] --> B[TaskScheduler] B --> C[(scheduled_tasks)] C --> D[OrderStateMachineTaskSpec] D --> E[StateMachineService] E --> F[状态机] subgraph "串行化执行" D E F end

核心思想：

  1. 所有事件通过 TaskScheduler 提交到 DB-Scheduler
  2. 每个事件创建一个唯一的 Task
  3. ShedLock 确保同一订单的事件串行执行
  4. 一个订单同一时刻只有一个线程操作状态机

8.3 最佳实践总结

1. 状态和事件使用枚举
类型安全，编译期检查，IDE 提示友好。

2. 业务逻辑放在 Action 中
状态机配置只负责流转逻辑，业务代码解耦到 Action。

3. 必须处理线程安全
使用 TaskSpec + DB-Scheduler 模式串行化访问。

4. 状态持久化
每次状态变更后立即持久化，防止重启丢失。

5. 监控状态转换
记录每次状态变更的日志，便于问题排查。
