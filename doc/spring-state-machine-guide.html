Spring State Machine 使用指南

一、简介

Spring State Machine 是 Spring 官方提供的状态机框架， 基于有限状态机（Finite State Machine）理论，帮助开发者在业务系统中实现复杂的状态流转逻辑。

🎯 声明式配置

通过 Builder API 或注解配置状态和转换

🔄 Barrier Aggregate 模式

支持等待多个屏障全部通过后进入下一状态

💾 状态持久化

JPA 持久化，支持 PostgreSQL JSONB

📢 事件驱动

与 Kafka 天然契合

二、线程安全架构

2.1 核心问题

Spring State Machine 不是线程安全的

多个线程同时操作同一个订单的状态机会导致状态混乱。
本项目通过 OrderCommandService → TaskSpec → ShedLock 保证线程安全。

2.2 架构设计

sequenceDiagram participant API as REST API participant Kafka as Kafka Consumer participant OCS as OrderCommandService participant TS as TaskScheduler participant DB as db-scheduler participant Task as OrderStateMachineTaskSpec participant Lock as ShedLock participant SM as StateMachine API->>OCS: submitOrderEvent(orderNo, event) Kafka->>OCS: onEvent(orderNo, event) OCS->>TS: submit(taskSpec, instanceId, payload) TS->>DB: INSERT scheduled_tasks DB->>Task: 异步执行任务 Task->>Lock: lock(orderNo) Lock-->>Task: 获取锁成功 Task->>SM: sendEvent(orderNo, event) SM-->>Task: 执行成功 Task->>Lock: unlock()

2.3 OrderCommandService 统一入口

OrderCommandService.kt
@Service
class OrderCommandService(
    private val taskScheduler: TaskScheduler,
    private val orderStateMachineTaskSpec: OrderStateMachineTaskSpec,
) {
    fun submitOrderEvent(orderNo: String, event: OrderEvent) {
        val instanceId = "order-$orderNo-${event.name}-${UUID.randomUUID()}"
        val payload = OrderEventPayload(orderNo, event)
        
        taskScheduler.submit(
            spec = orderStateMachineTaskSpec,
            instanceId = instanceId,
            payload = payload,
        )
    }
}

三、状态机配置

3.1 StateMachineConfig 核心配置

StateMachineConfig.kt
@Configuration
@EnableStateMachineFactory
class StateMachineConfig(
    private val listener: StateMachineListener,
) : StateMachineConfigurerAdapter<OrderStatus, OrderEvent>() {
    
    override fun configure(states: StateMachineStateConfigurer<OrderStatus, OrderEvent>) {
        states
            .withStates()
            .initial(OrderStatus.INIT)
            .states(EnumSet.allOf(OrderStatus::class.java))
    }
    
    override fun configure(transitions: StateMachineTransitionConfigurer<OrderStatus, OrderEvent>) {
        // INIT → LOCAL_INITIALIZED → FACTORY_ORDER_SUBMITTED
        transitions
            .withExternal()
                .source(OrderStatus.INIT)
                .target(OrderStatus.LOCAL_INITIALIZED)
                .event(OrderEvent.PR_APPROVED)
                .action(prApprovedAction)
            .and()
            .withExternal()
                .source(OrderStatus.LOCAL_INITIALIZED)
                .target(OrderStatus.FACTORY_ORDER_SUBMITTED)
                .action(initializeOrderInitBarrierAction)
            // ... 更多转换配置
    }
    
    override fun configure(config: StateMachineConfigurationConfigurer<OrderStatus, OrderEvent>) {
        config.withConfiguration()
            .listener(listener)  // 全局注册 Listener
            .autoStartup(false)
    }
}

3.2 状态同步机制

StateMachineListener.kt
@Component
class StateMachineListener(
    private val orderJpaRepository: OrderJpaRepository,
    private val transactionTemplate: TransactionTemplate,
) : StateMachineListenerAdapter<OrderStatus, OrderEvent>() {
    
    // 关键：使用 STATE_CHANGED 阶段获取状态机 ID
    // Spring State Machine 4.x 的 State 接口没有 stateMachine 属性
    override fun stateChanged(
        from: State<OrderStatus, OrderEvent>?,
        to: State<OrderStatus, OrderEvent>?,
        context: StateContext<OrderStatus, OrderEvent>,
    ) {
        if (context.stage == StateContext.Stage.STATE_CHANGED) {
            val orderNo = context.stateMachine.id
            val newStatus = to?.id ?: return
            
            // 使用 TransactionTemplate 确保 Spring AOP 生效
            transactionTemplate.execute {
                val entity = orderJpaRepository.findByOrderNo(orderNo) ?: return@execute
                entity.status = newStatus
                entity.updatedAt = Instant.now()
            }
        }
    }
}

Spring AOP 不拦截同类内方法调用

不能在 Listener 内部调用带 @Transactional 的方法，需要使用 TransactionTemplate。

四、Barrier Aggregate 模式

4.1 模式说明

为什么需要 Barrier Aggregate？

某些状态转换需要等待多个外部事件全部到达后才能进行。
例如：FACTORY_ORDER_SUBMITTED 状态需要等待 VOM 和 DOM 两个事件。

4.2 数据模型

BarrierAggregateRecord.kt
@Entity
@Table(name = "barrier_aggregate")
class BarrierAggregateRecord(
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    val id: Long? = null,
    
    @Column(name = "aggregate_type", nullable = false)
    val aggregateType: String,
    
    @Column(name = "aggregate_key", nullable = false)
    val aggregateKey: String,  // orderNo
    
    @Column(name = "required_barriers", columnDefinition = "jsonb")
    @Convert(converter = BarrierListConverter::class)
    var requiredBarriers: List<String>,
    
    @Column(name = "passed_barriers", columnDefinition = "jsonb")
    @Convert(converter = BarrierListConverter::class)
    var passedBarriers: MutableList<String> = mutableListOf(),
    
    @Version
    var version: Long = 0,
)

4.3 MarketAwareBarrierAggregate

MarketAwareBarrierAggregate.kt
abstract class MarketAwareBarrierAggregate(
    repository: BarrierAggregateRepository,
    private val barrierType: BarrierType,
) : BarrierAggregate(repository) {
    
    protected abstract val marketBarrierConfig: Map<Market, List<String>>
    
    override fun getRequiredBarriers(market: Market): List<String> {
        return marketBarrierConfig[market] ?: emptyList()
    }
    
    abstract override fun onAllBarriersPassed(aggregateKey: String)
    
    abstract override fun onBarrierFailed(aggregateKey: String)
}

4.4 具体实现

PurchaseRequestAcceptBarrierAggregate.kt
@Component
class PurchaseRequestAcceptBarrierAggregate(
    repository: BarrierAggregateRepository,
    private val orderCommandService: OrderCommandService,
) : MarketAwareBarrierAggregate(repository, PurchaseRequestAcceptBarrier) {
    
    override val marketBarrierConfig = mapOf(
        Market.DE to listOf("SVS", "PRICE", "FINANCE"),
        Market.IT to listOf("SVS", "BODYBUILDER", "CONTRACT_ROLES", "PRICING", "PAYMENT_SPLIT", "FINANCING_BLUEPRINT"),
    )
    
    override fun onAllBarriersPassed(aggregateKey: String) {
        orderCommandService.submitOrderEvent(aggregateKey, OrderEvent.PURCHASE_REQUEST_ACCEPT_SUCCESS)
    }
    
    override fun onBarrierFailed(aggregateKey: String) {
        orderCommandService.submitOrderEvent(aggregateKey, OrderEvent.PURCHASE_REQUEST_ACCEPT_FAILED)
    }
}

五、Action 设计

5.1 AbstractBroadcastAction

AbstractBroadcastAction.kt
abstract class AbstractBroadcastAction(
    private val orderEventProducer: OrderEventProducer,
) : Action<OrderStatus, OrderEvent> {
    
    protected abstract val targetStatus: OrderStatus
    
    override fun execute(context: StateContext<OrderStatus, OrderEvent>) {
        val orderNo = OrderActionUtils.extractOrderNo(context)
        orderEventProducer.broadcastStatusChange(orderNo, targetStatus)
    }
}

5.2 OrderActionUtils

OrderActionUtils.kt
object OrderActionUtils {
    fun extractOrderNo(context: StateContext<OrderStatus, OrderEvent>): String {
        return context.message.headers.get("orderNo") as? String
            ?: context.stateMachine.id
            ?: throw IllegalStateException("Cannot extract orderNo from context")
    }
}

六、测试策略

6.1 自动转换状态验证

问题场景

PR_APPROVED event received
└─> INIT → LOCAL_INITIALIZED (有事件的转换)
    └─> LOCAL_INITIALIZED → FACTORY_ORDER_SUBMITTED (自动转换)

STATE_CHANGED 回调顺序可能与转换顺序相反。

6.2 测试策略

1. 验证 Action 执行结果
检查 Action 的副作用，如 barrier aggregate 是否创建

2. 验证最终状态
跳过中间状态，直接验证有事件触发的目标状态

测试示例
// 不验证中间状态
// assertEquals(OrderStatus.FACTORY_ORDER_SUBMITTED, order.status)

// 改为验证 Action 执行结果
await().untilAsserted {
    val barriers = barrierAggregateJpaRepository.findAll()
    assertTrue(barriers.any { it.aggregateKey == orderNo })
}

七、最佳实践总结

1. 使用 orderNo 作为 machineId - 外部系统友好

2. 统一使用 OrderCommandService - 保证线程安全

3. 状态同步使用 TransactionTemplate - 避免 Spring AOP 问题

4. Barrier Aggregate 使用乐观锁 - 防止并发问题

5. 测试时跳过中间状态验证 - 避免 Spring State Machine 回调顺序不确定性
