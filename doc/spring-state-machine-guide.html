Spring State Machine 使用指南

一、简介

Spring State Machine 是 Spring 官方提供的状态机框架， 基于有限状态机（Finite State Machine）理论，帮助开发者在业务系统中实现复杂的状态流转逻辑。

🎯 声明式配置

通过 Builder API 或注解配置状态和转换

🔄 Fork/Join 模式

支持并行等待多个事件

💾 状态持久化

JPA、Redis 等多种持久化方案

📢 事件驱动

与 Kafka 天然契合

二、状态机绑定设计

2.1 使用 orderNo 作为 machineId

核心设计：使用 orderNo（业务订单号）作为状态机的 machineId

• 外部系统天然知道 orderNo，无需查询 orderId
• orderNo 是自然主键，避免 ID 迁移问题
• 调试友好，machineId = "ORD-001" 直观可读 

订单表 (orders)              状态机表 (state_machine)
┌────────────────┐           ┌──────────────────┐
│ id (PK)        │           │ machine_id (PK)  │
│ order_no ──────┼───────────┼─► "ORD-001"      │
│ status         │           │ state            │
└────────────────┘           └──────────────────┘

绑定关系: orders.order_no = state_machine.machine_id

2.2 StateMachineService 核心 API

StateMachineService.kt
@Service
class StateMachineService(
    private val stateMachineFactory: StateMachineFactory<OrderStatus, OrderEvent>,
    private val jpaStateMachineRepository: JpaStateMachineRepository,
) {
    /**
     * 发送事件（使用 orderNo 作为 machineId）
     * 这是推荐的 API
     */
    fun sendEventByOrderNo(orderNo: String, event: OrderEvent, headers: Map<String, Any> = emptyMap()): Boolean {
        val machineId = orderNo  // 关键：machineId = orderNo
        val stateMachine = stateMachineFactory.getStateMachine(machineId)
        
        restoreStateMachine(stateMachine, machineId).block()
        stateMachine.startReactively().block()
        
        val message = MessageBuilder.withPayload(event)
            .setHeader("orderNo", orderNo)
            .apply { headers.forEach { (k, v) -> setHeader(k, v) } }
            .build()
        
        val result = stateMachine.sendEvent(Mono.just(message))
            .next().map { it.resultType == ResultType.ACCEPTED }.block() ?: false
        
        if (result) persistStateMachine(stateMachine, machineId)
        stateMachine.stopReactively().block()
        
        return result
    }
}

三、事务处理策略

3.1 核心问题

Spring State Machine 不是事务安全的

• 状态机操作涉及 Reactive 流
• Action 执行需要与数据库事务协调
• 状态同步需要独立事务

3.2 事务边界设计

组件 事务策略 原因
StateMachineService.sendEventByOrderNo() 无事务 涉及 Reactive 流
Action.execute() 无 @Transactional 由 Repository 方法继承事务
OrderStatusSyncService.syncOrderStatusByOrderNo() @Transactional(REQUIRES_NEW) 独立事务，确保隔离

3.3 状态同步：REQUIRES_NEW 策略

为什么使用 REQUIRES_NEW？

1. 隔离性：状态同步失败不影响状态机持久化
2. 一致性：即使外部事务回滚，状态同步也已提交
3. 错误处理：catch 异常后仅记录日志，不传播

OrderStatusSyncService.kt
@Service
class OrderStatusSyncService(private val orderJpaRepository: OrderJpaRepository) {
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    fun syncOrderStatusByOrderNo(orderNo: String, newStatus: OrderStatus): Boolean {
        val entity = orderJpaRepository.findByOrderNo(orderNo) ?: return false
        entity.status = newStatus
        entity.updatedAt = Instant.now()
        orderJpaRepository.save(entity)
        return true
    }
}

3.4 状态同步触发时机

sequenceDiagram participant SM as StateMachineService participant StateMachine participant SML as StateMachineListener participant OSS as OrderStatusSyncService participant DB as orders 表 SM->>StateMachine: sendEventByOrderNo(orderNo, event) StateMachine->>StateMachine: 恢复状态 / 启动 / 发送事件 StateMachine->>StateMachine: persistStateMachine() StateMachine->>StateMachine: stopReactively() StateMachine->>SML: stateMachineStopped(machineId) SML->>OSS: syncOrderStatusByOrderNo(orderNo, finalState) Note over OSS: @Transactional(REQUIRES_NEW) OSS->>DB: UPDATE orders SET status = ?
StateMachineListener.kt
@Component
class StateMachineListener(private val orderStatusSyncService: OrderStatusSyncService) 
    : StateMachineListenerAdapter<OrderStatus, OrderEvent>() {
    
    override fun stateMachineStopped(stateMachine: StateMachine<OrderStatus, OrderEvent>) {
        val orderNo = stateMachine.id  // machineId = orderNo
        val finalState = stateMachine.state?.id ?: return
        
        try {
            orderStatusSyncService.syncOrderStatusByOrderNo(orderNo, finalState)
        } catch (e: Exception) {
            log.error("Status sync failed but state machine continues", e)
        }
    }
}

3.5 Action 事务处理

PrApprovedAction.kt - 幂等处理
@Component
class PrApprovedAction(
    private val orderRepository: OrderRepository,
    private val stateMachineService: StateMachineService,
) : Action<OrderStatus, OrderEvent> {
    
    override fun execute(context: StateContext<OrderStatus, OrderEvent>) {
        // 无 @Transactional，由 Repository 方法继承事务
        val orderNo = context.message.headers.get("orderNo") as? String ?: return
        
        // 幂等检查
        val existingOrder = orderRepository.findByOrderNo(orderNo)
        if (existingOrder != null) {
            val existingStatus = stateMachineService.getCurrentStateByOrderNo(orderNo)
            if (existingStatus != null && existingStatus != OrderStatus.INIT) {
                log.info("Order already processed, skipping: orderNo=$orderNo")
                return
            }
        }
        
        // 创建新订单
        val order = Order.fromPrApproved(orderNo, ...)
        orderRepository.save(order)  // 继承事务
        stateMachineService.initializeStateMachineByOrderNo(orderNo, OrderStatus.LOCAL_INITIALIZED)
    }
}

3.6 事务最佳实践总结

1. StateMachineService 方法不加 @Transactional
Reactive 流不能在传统事务内执行

2. Action 不加 @Transactional
由 Repository 继承事务，或调用独立事务服务

3. 状态同步使用 REQUIRES_NEW
独立事务，失败不影响状态机

4. 状态同步放在 stateMachineStopped
确保每次状态变更都同步到业务表

四、核心概念

组件 说明 项目中的定义
State 状态，枚举类型 OrderStatus：INIT, LOCAL_INITIALIZED, ...
Event 事件，触发状态转换 OrderEvent：PR_APPROVED, VOM, DOM, ...
Action 动作，转换时执行的逻辑 PrApprovedAction, SendCoeAction, ...
StateMachine 状态机实例 每个订单一个实例，通过 orderNo 标识
