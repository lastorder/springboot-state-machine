Spring State Machine 使用指南

一、简介

Spring State Machine 是 Spring 官方提供的状态机框架， 基于有限状态机（Finite State Machine）理论，帮助开发者在业务系统中实现复杂的状态流转逻辑。 它提供了声明式的配置方式、丰富的状态转换机制、以及与 Spring 生态的深度集成。

🎯 声明式配置

通过 Builder API 或注解配置状态和转换，代码清晰易维护

🔄 丰富的转换模式

支持外部转换、内部转换、Fork/Join、Entry/Exit Action 等

💾 状态持久化

提供 JPA、Redis、MongoDB 等多种持久化方案

🔐 Guard 条件

转换前可配置守卫条件，实现复杂的业务校验

📢 事件驱动

事件触发的状态转换，与消息队列天然契合

🧪 测试支持

提供 StateMachineTestPlan 简化测试

二、解决的问题

2.1 为什么需要状态机？

在业务系统中，状态管理是一个常见但容易出错的问题：

传统的方式（if-else / switch）： if (status == "CREATED") { if (event == "PAY") { // ❌ 非法转换！订单还未确认 } else if (event == "CONFIRM") { status = "PENDING_PAYMENT" } } else if (status == "PENDING_PAYMENT") { if (event == "PAY") { status = "PAID" } } ... // 😱 难以维护！ 问题： 1. 状态转换逻辑分散，没有统一入口 2. 非法转换难以防止 3. 状态图谱不直观 4. 测试困难

Spring State Machine 提供了更优雅的解决方案：

  • 集中配置：所有状态和转换在一处定义，清晰可见
  • 类型安全：使用枚举定义状态和事件，编译期检查
  • 自动校验：非法转换会被自动拒绝
  • 可视化：状态图谱一目了然

2.2 本项目的业务场景

本项目是一个订单管理系统，需要处理复杂的状态流转：

核心需求

1. 订单创建后需要进行 并行验证（库存检查 + 价格检查）
2. 验证成功后等待 用户确认
3. 用户确认后进入 支付流程
4. 任意环节可 取消，完成后可 退款

三、核心概念

3.1 基础组件

graph LR A[State] --> B[StateMachine] C[Event] --> B B --> D[Transition] D --> E[Action] D --> F[Guard] subgraph "状态机核心" B end
组件 说明 项目中的定义
State 状态，枚举类型 OrderStatus：CREATED, PENDING_PAYMENT, PAID...
Event 事件，触发状态转换 OrderEvent：PAY, CANCEL, CONFIRM...
Transition 转换，定义 State → State 的路径 source=PENDING_PAYMENT, target=PAID, event=PAY
Action 动作，转换时执行的逻辑 PaymentAction：处理支付逻辑
Guard 守卫，转换前的条件判断 PaymentGuard：检查支付条件
StateMachine 状态机实例 每个订单一个实例，通过 orderId 标识

3.2 状态分类

类型 说明 项目中的使用
Initial State 初始状态，状态机启动时进入 CREATED
End State 终止状态，状态机停止 DELIVERED, CANCELLED, REJECTED, REFUNDED
Fork State 分支状态，进入多个并行区域 PENDING_VALIDATION
Join State 汇聚状态，等待所有并行区域完成 PENDING_CONFIRMATION
Parent State 父状态，包含子状态 PENDING_VALIDATION（包含 INVENTORY_CHECK, PRICING_CHECK）

4.3 配置状态机

StateMachineConfig.kt
@Configuration
@EnableStateMachineFactory
class StateMachineConfig(
    private val paymentAction: PaymentAction,
    private val paymentGuard: PaymentGuard,
    // ... other actions
) : StateMachineConfigurerAdapter<OrderStatus, OrderEvent>() {
    
    override fun configure(states: StateMachineStateConfigurer<OrderStatus, OrderEvent>) {
        states
            .withStates()
            .initial(OrderStatus.CREATED)
            .fork(OrderStatus.PENDING_VALIDATION)
            .join(OrderStatus.PENDING_CONFIRMATION)
            .state(OrderStatus.PENDING_CONFIRMATION)
            .state(OrderStatus.PENDING_PAYMENT)
            .end(OrderStatus.DELIVERED)
            .end(OrderStatus.CANCELLED)
            // ... more states
            .and()
            .withStates()
            .parent(OrderStatus.PENDING_VALIDATION)
            .initial(OrderStatus.INVENTORY_CHECK)
            .state(OrderStatus.INVENTORY_CHECK, inventoryCheckAction, null)
            .and()
            .withStates()
            .parent(OrderStatus.PENDING_VALIDATION)
            .initial(OrderStatus.PRICING_CHECK)
            .state(OrderStatus.PRICING_CHECK, pricingCheckAction, null)
    }
    
    override fun configure(transitions: StateMachineTransitionConfigurer<OrderStatus, OrderEvent>) {
        transitions
            .withExternal()
            .source(OrderStatus.PENDING_PAYMENT)
            .target(OrderStatus.PAID)
            .event(OrderEvent.PAY)
            .guard(paymentGuard)
            .action(paymentAction)
            .and()
            // ... more transitions
    }
}

4.4 状态配置详解

states
    .withStates()
    .initial(OrderStatus.CREATED)           // 初始状态
    .end(OrderStatus.DELIVERED)             // 终止状态
    .fork(OrderStatus.PENDING_VALIDATION)   // Fork 状态
    .join(OrderStatus.PENDING_CONFIRMATION) // Join 状态
    .state(OrderStatus.PENDING_PAYMENT)     // 普通状态
    
    // 带 Entry/Exit Action 的状态
    .state(OrderStatus.INVENTORY_CHECK, entryAction, exitAction)
    
    // 子状态区域
    .and()
    .withStates()
    .parent(OrderStatus.PENDING_VALIDATION) // 父状态
    .initial(OrderStatus.INVENTORY_CHECK)   // 子区域初始状态

4.5 转换配置详解

转换类型 说明 使用场景
withExternal() 外部转换：Source → Target 大多数状态变更
withInternal() 内部转换：状态不变，执行 Action 状态内的业务逻辑
withFork() Fork 转换：进入并行区域 并行处理
withJoin() Join 转换：汇聚并行区域 等待并行完成
外部转换示例
transitions
    // 外部转换：状态的真正变更
    .withExternal()
    .source(OrderStatus.PENDING_PAYMENT)
    .target(OrderStatus.PAID)
    .event(OrderEvent.PAY)
    .guard(paymentGuard)      // 可选：守卫条件
    .action(paymentAction)    // 可选：执行动作
    .and()
    
    // 内部转换：状态不变，只执行 Action
    .withInternal()
    .source(OrderStatus.PENDING_CONFIRMATION)
    .event(OrderEvent.INVENTORY_MODIFIED)
    .action(notifyAction)
    .and()
    
    // Fork 转换：进入并行区域
    .withFork()
    .source(OrderStatus.CREATED)
    .target(OrderStatus.PENDING_VALIDATION)
    .and()
    
    // Join 转换：汇聚多个并行状态
    .withJoin()
    .source(OrderStatus.INVENTORY_CHECK)
    .source(OrderStatus.PRICING_CHECK)
    .target(OrderStatus.PENDING_CONFIRMATION)

4.6 Action 编写

PaymentAction.kt
@Component
class PaymentAction(
    private val orderRepository: OrderRepository,
    private val paymentService: PaymentService,
) : Action<OrderStatus, OrderEvent> {
    
    private val log = LoggerFactory.getLogger(javaClass)
    
    override fun execute(context: StateContext<OrderStatus, OrderEvent>) {
        // 1. 从消息头获取 orderId
        val orderId = context.message.headers["orderId", Long::class.java]
            ?: return
        
        // 2. 执行业务逻辑
        val order = orderRepository.findById(orderId) ?: return
        paymentService.processPayment(order)
        
        // 3. 更新数据库
        orderRepository.save(order)
        
        log.info("Payment processed: orderId=$orderId")
    }
}

4.7 Guard 编写

PaymentGuard.kt
@Component
class PaymentGuard(
    private val orderRepository: OrderRepository,
) : Guard<OrderStatus, OrderEvent> {
    
    override fun evaluate(context: StateContext<OrderStatus, OrderEvent>): Boolean {
        val orderId = context.message.headers["orderId", Long::class.java]
            ?: return false
        
        val order = orderRepository.findById(orderId) ?: return false
        
        // 检查支付条件
        return order.amount > BigDecimal.ZERO
    }
}

Guard vs Action

Guard：在转换 之前 执行，返回 false 则转换被拒绝
Action：在转换 过程中 执行，不会阻止转换

用 Guard 做 前置校验，用 Action 做 业务处理。

五、Fork/Join 并行模式

5.1 什么是 Fork/Join？

Fork/Join 是一种并行处理模式：Fork 将流程拆分为多个并行分支， Join 等待所有分支完成后汇聚。

stateDiagram-v2 [*] --> CREATED CREATED --> PENDING_VALIDATION : SUBMIT_VALIDATION state PENDING_VALIDATION { state fork_state <<fork>> state join_state <<join>> [*] --> fork_state fork_state --> INVENTORY_CHECK fork_state --> PRICING_CHECK INVENTORY_CHECK --> join_state : INVENTORY_SUCCESS PRICING_CHECK --> join_state : PRICING_SUCCESS join_state --> [*] } PENDING_VALIDATION --> CANCELLED : INVENTORY_FAILED PENDING_VALIDATION --> CANCELLED : PRICING_FAILED PENDING_VALIDATION --> PENDING_CONFIRMATION : Join success

5.2 项目中的实现

状态定义

父状态：PENDING_VALIDATION ├── 子状态1：INVENTORY_CHECK（库存检查） └── 子状态2：PRICING_CHECK（价格检查） 流程： 1. CREATED ──Fork──▶ PENDING_VALIDATION 2. PENDING_VALIDATION ──进入──▶ INVENTORY_CHECK + PRICING_CHECK（并行） 3. INVENTORY_CHECK ◀──INVENTORY_SUCCESS── 完成 4. PRICING_CHECK ◀──PRICING_SUCCESS── 完成 5. 两个都完成 ──Join──▶ PENDING_CONFIRMATION

配置代码

StateMachineConfig.kt
override fun configure(states: StateMachineStateConfigurer<OrderStatus, OrderEvent>) {
    states
        .withStates()
        .initial(OrderStatus.CREATED)
        .fork(OrderStatus.PENDING_VALIDATION)        // Fork 状态
        .join(OrderStatus.PENDING_CONFIRMATION)      // Join 状态
        .and()
        
        // 子状态区域1：库存检查
        .withStates()
        .parent(OrderStatus.PENDING_VALIDATION)      // 属于父状态
        .initial(OrderStatus.INVENTORY_CHECK)        // 子区域初始状态
        .end(OrderStatus.INVENTORY_CHECK)            // 子区域结束状态
        .state(OrderStatus.INVENTORY_CHECK, inventoryCheckAction, null)
        .and()
        
        // 子状态区域2：价格检查
        .withStates()
        .parent(OrderStatus.PENDING_VALIDATION)
        .initial(OrderStatus.PRICING_CHECK)
        .end(OrderStatus.PRICING_CHECK)
        .state(OrderStatus.PRICING_CHECK, pricingCheckAction, null)
}

override fun configure(transitions: StateMachineTransitionConfigurer<OrderStatus, OrderEvent>) {
    transitions
        // Fork 转换
        .withFork()
        .source(OrderStatus.CREATED)
        .target(OrderStatus.PENDING_VALIDATION)
        .and()
        
        // Join 转换
        .withJoin()
        .source(OrderStatus.INVENTORY_CHECK)    // 来自子状态1
        .source(OrderStatus.PRICING_CHECK)      // 来自子状态2
        .target(OrderStatus.PENDING_CONFIRMATION) // 汇聚目标
}

5.3 并行验证的处理

库存和价格检查通过 Kafka 异步返回结果：

sequenceDiagram participant SM as StateMachine participant Action as ValidationSubmitAction participant Kafka as Kafka participant Inventory as 库存服务 participant Pricing as 价格服务 SM->>Action: execute() Action->>Kafka: inventory.check.request Action->>Kafka: pricing.request Note over Action: 并行发送两个请求 par 并行处理 Kafka->>Inventory: 处理请求 Inventory-->>Kafka: inventory.check.response and Kafka->>Pricing: 处理请求 Pricing-->>Kafka: pricing.response end Kafka-->>SM: INVENTORY_SUCCESS / INVENTORY_FAILED Kafka-->>SM: PRICING_SUCCESS / PRICING_FAILED alt 两个都成功 SM->>SM: Join → PENDING_CONFIRMATION else 任一失败 SM->>SM: → CANCELLED end

5.4 Join 条件

Join 的触发条件

Spring State Machine 的 Join 需要所有并行区域都到达自己的结束状态。

本项目中：
• INVENTORY_CHECK 需要收到 INVENTORY_SUCCESS 才能到达结束状态
• PRICING_CHECK 需要收到 PRICING_SUCCESS 才能到达结束状态
• 两者都结束后，自动触发 Join，进入 PENDING_CONFIRMATION

六、状态持久化

6.1 为什么需要持久化？

Spring State Machine 默认将状态保存在内存中。但在实际业务中：

  • 应用重启后状态丢失
  • 集群环境下状态不同步
  • 需要查询历史状态

通过 JPA 持久化，状态机状态存储在数据库中。

6.3 使用持久化

StateMachineService.kt
@Service
class StateMachineService(
    private val stateMachineFactory: StateMachineFactory<OrderStatus, OrderEvent>,
    private val jpaStateMachineRepository: JpaStateMachineRepository,
) {
    fun sendEvent(orderId: Long, event: OrderEvent, headers: Map<String, Any>): Boolean {
        val machineId = orderId.toString()
        val stateMachine = stateMachineFactory.getStateMachine(machineId)
        
        return try {
            // 1. 从数据库恢复状态
            restoreStateMachine(stateMachine, machineId)
            
            // 2. 发送事件
            val message = MessageBuilder
                .withPayload(event)
                .setHeader("orderId", orderId)
                .apply { headers.forEach { (k, v) -> setHeader(k, v) } }
                .build()
            
            val result = stateMachine
                .sendEvent(Mono.just(message))
                .next()
                .map { it.resultType == ResultType.ACCEPTED }
                .block() ?: false
            
            // 3. 持久化状态
            if (result) {
                persistStateMachine(stateMachine, machineId)
            }
            
            result
        } finally {
            stateMachine.stopReactively().block()
        }
    }
    
    private fun restoreStateMachine(stateMachine: StateMachine<OrderStatus, OrderEvent>, machineId: String) {
        val entity = jpaStateMachineRepository.findById(machineId)
        if (entity.isPresent) {
            val state = OrderStatus.valueOf(entity.get().state)
            stateMachine.stateMachineAccessor.doWithRegion { region ->
                region.resetStateMachine(
                    DefaultStateMachineContext(state, null, null, null, null, machineId)
                )
            }
        }
    }
    
    private fun persistStateMachine(stateMachine: StateMachine<OrderStatus, OrderEvent>, machineId: String) {
        val entity = JpaRepositoryStateMachine()
        entity.machineId = machineId
        entity.state = stateMachine.state.id.name
        jpaStateMachineRepository.save(entity)
    }
}

七、线程安全问题

7.1 问题分析

Spring State Machine 不是线程安全的！

官方文档明确指出：一个 StateMachine 实例不应该被多个线程同时访问。

原因：
• 状态机内部使用非线程安全的数据结构
• 状态变更过程中存在竞态条件
• 并发操作可能导致状态不一致

7.2 风险场景

sequenceDiagram participant T1 as 线程1 participant T2 as 线程2 participant SM as StateMachine participant DB as Database Note over SM: 当前状态: PENDING_PAYMENT par 并发请求 T1->>SM: sendEvent(PAY) SM->>SM: 检查状态: PENDING_PAYMENT SM->>SM: 转换中... and T2->>SM: sendEvent(CANCEL) SM->>SM: 检查状态: PENDING_PAYMENT (旧值!) end Note over SM: 状态混乱！ SM-->>T1: ACCEPTED SM-->>T2: ACCEPTED Note over DB: 最终状态可能是 PAID 或 CANCELLED
取决于执行顺序

7.3 解决方案：Command Inbox + DB-Scheduler

本项目采用 Command Inbox Pattern 解决线程安全问题：

graph LR A[HTTP/Kafka] --> B[Command Inbox] B --> C[(Database)] C --> D[DB-Scheduler] D --> E[StateMachine] subgraph "串行化执行" D E end

核心思想：

  1. 所有状态变更请求写入 Command Inbox（数据库表）
  2. DB-Scheduler 为每个订单创建唯一的 Task
  3. Task 串行处理同一订单的所有命令
  4. 一个订单同一时刻只有一个线程操作状态机

One Task Per Order

DB-Scheduler 使用 SELECT FOR UPDATE 确保同一 task_id 只有一个线程执行。

设置 task_id = orderId，确保每个订单的命令串行处理。

8.3 最佳实践总结

1. 状态和事件使用枚举
类型安全，编译期检查，IDE 提示友好。

2. 业务逻辑放在 Action 中
状态机配置只负责流转逻辑，业务代码解耦到 Action。

3. 使用 Guard 做前置校验
不应该在 Guard 中执行有副作用的操作。

4. 必须处理线程安全
使用 Command Inbox + DB-Scheduler 模式串行化访问。

5. 状态持久化
每次状态变更后立即持久化，防止重启丢失。

6. 监控状态转换
记录每次状态变更的日志，便于问题排查。

8.4 常见问题

Q1：为什么不直接在 Action 中操作数据库？

Action 在状态机内部执行，事务处理复杂。建议 Action 只做轻量操作， 重业务逻辑放在 Service 层，Action 调用 Service。

Q2：状态机实例是单例还是原型？

使用 @EnableStateMachineFactory 时，每次调用 getStateMachine(id) 返回新实例。实例间通过 machineId 和数据库共享状态。

Q3：如何处理转换失败？

事件发送返回 false 表示转换被拒绝（Guard 返回 false 或非法转换）。 业务上应该处理这种情况，本项目中会将命令标记为 SKIPPED。

Q4：如何取消正在进行的 Join？

Join 过程中，可以发送外部事件（如 CANCEL）从父状态转换出去， 这会中断 Join 的等待。本项目中 INVENTORY_FAILED 就是这样处理的。
